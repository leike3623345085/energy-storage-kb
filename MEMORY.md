## 🎯 核心原则

### 执行准则
**遇到问题直接执行最优解决方案，事后通报结果** —— 不反复询问确认。

| 场景 | 行动 |
|------|------|
| 发现问题 | 立即分析 → 执行修复 → 通报结果 |
| 多种方案 | 选择最优直接执行，不逐一请示 |
| 需要确认 | 仅在涉及安全风险或重大变更时询问 |

### 执行准则细节
1. **先行动后汇报** — 修复完成再告知用户处理结果
2. **默认自主** — 常规运维问题自行决策执行
3. **记录留痕** — 所有操作自动保存到经验教训
4. **出错修复直接处理** — 发现任务执行错误（如 cron 任务失败、脚本语法错误等），直接分析并修复，无需事先询问用户，修复后通报结果即可

## 💾 记忆自动保存机制（2026-03-23 启用）

### 触发条件
| 内容类型 | 检测方式 | 保存位置 |
|---------|---------|----------|
| 代码/脚本 | 检测代码块 (```language) | `memory/auto/snippet_*.py` |
| 用户偏好 | 检测"记住/以后/默认"等关键词 | `USER.md` |
| 执行规则 | 检测决策/准则类内容 | `MEMORY.md` |
| 经验教训 | 检测"问题/修复/bug/解决" | `memory/lessons/{project}.md` |
| 项目技术 | 检测"架构/技术/实现/模块" | `memory/system/{project}.md` |
| 对话归档 | 每6小时自动运行 | `memory/auto/YYYY-MM-DD_auto.md` |

### 自动任务
- **定时归档**: 每6小时自动分析会话，提取关键信息归档
- **cron ID**: `91fa57f8-460b-4f16-b6e8-4f14ec06bfbb`

### 记忆冲突解决机制
当新记忆与旧记忆矛盾时，按以下策略处理：

| 冲突类型 | 检测方式 | 解决策略 | 用户参与 |
|---------|---------|---------|---------|
| **直接矛盾** | "喜欢X" vs "不喜欢X" | 暂停并询问用户 | 必须确认 |
| **更新细化** | 相似度50-80% | 自动合并，事后通知 | 无需确认 |
| **版本覆盖** | 相似度<50% | 新记忆优先，记录日志 | 无需确认 |
| **规则变更** | 执行准则变化 | 明确标记版本时间 | 事后通报 |

**冲突解决原则：**
1. **高冲突**（直接矛盾）：暂停保存，询问用户"覆盖/保留/合并"
2. **中冲突**（更新细化）：自动合并，保留历史版本，事后通报
3. **低冲突**（自然演变）：新记忆优先覆盖，记录变更日志

**时间戳策略：**
- 所有记忆条目必须包含时间戳
- 旧记忆不删除，标记为【历史版本】
- 允许同一主题存在多个时间版本

### 记忆激活机制（确保保存的记忆真正被使用）

**问题：** 保存了但读取效率不高

**解决方案：**

| 场景 | 激活机制 | 实现方式 |
|------|---------|---------|
| **会话启动** | 自动回顾最近记忆 | `context_loader.py` 生成摘要 |
| **任务执行前** | 关键词触发搜索 | 检测到"之前/上次/项目"→ 自动 memory_search |
| **保存时** | 建立关联索引 | 保存时记录相关关键词，便于后续召回 |

**会话启动检查清单：**
1. ✅ 读取 SOUL.md + USER.md（已有）
2. ✅ 读取今日日志（已有）
3. 🆕 **生成记忆摘要**（最近项目/教训/归档）
4. 🆕 **准备搜索关键词**（根据用户输入动态检索）

**使用规则：**
- 用户提到"之前/上次/记得"→ 必须调用 memory_search
- 涉及具体项目（储能/银行）→ 先读 memory/system/ 对应文件
- 涉及问题修复 → 先读 memory/lessons/ 对应文件

### 脚本位置
- `memory/auto_saver.py` - 核心保存逻辑
- `memory/save_trigger.py` - 触发器（对话中调用）
- `memory/auto_archive.py` - 定期归档任务
- `memory/conflict_resolver.py` - 冲突检测与解决

## 🏗️ Harness Engineering 架构规范

### 核心原则
**所有工作流都必须复用 Harness Engineering 架构**

> "人类掌舵，智能体执行" — Humans steer. Agents execute.

### 新建工作流必须包含的组件
- **Guardrails** — 飞行前检查、输出验证
- **Feedback Loop** — 错误分类、自动修复、规则学习
- **Progressive Disclosure** — 分层上下文加载
- **Agent Runner** — 总控协调

### 禁止事项
- ❌ 直接调用 AI 生成而不经过 Guardrails 验证
- ❌ 忽略错误处理，不接入 Feedback Loop
- ❌ 一次性加载所有上下文，不使用 Progressive Disclosure
- ❌ 重复造轮子，不复用 harness/ 已有组件

### 工作流模板位置
- `energy_storage/harness/README.md` - 架构文档
- `energy_storage/harness/harness_config.yaml` - 配置参考

## ⚙️ 定时任务通用规范

### Delivery 配置
- `mode: "announce"` 必须配合 `to: "main"` 使用
- 不需要推送通知的任务使用 `mode: "none"`

### 超时设置
- 简单脚本（读取文件、执行命令）：60 秒
- 网络请求任务：120-300 秒
- 复杂分析任务（需要 LLM 处理）：300-600 秒

### 任务设计原则
1. **脚本化优先** — 复杂逻辑写成 Python 脚本
2. **数据本地处理** — 避免在 Agent session 中传输大量数据
3. **幂等性** — 任务应能安全地重复执行
4. **错误隔离** — 使用 `sessionTarget: "isolated"`
